# FunkCode interpreter — structural decode

The bytecode interpreter `FUN_00472bc0` (va 0x00472bc0) is structurally
understood. Full operand decoding for every opcode is not yet complete, but the
dispatcher's calling convention, record layout, opcode partition and ~50
specific opcode encodings are firm.

## Record layout

From caller `FUN_00491170` (xref 1 of 19): `local_188 = 4; FUN_00472bc0(&local_188, buf)`.
So `param_1 = &ip` (initial value 4), `param_2 = buffer`. Combined with the
existing record framing from `funkcode_walker.py`:

```
+0   tag       u8      record type
+1   size_hi   u8   ┐
+2   size_lo   u8   ┘  u16 BE — total record length including this header
+3   flags     u8       NOT executed — pre-amble byte
+4   opcode_0  u8       <-- ip starts here
+5   ...
+N   opcode_k         where N+1 = size from header
```

Size check `*(ushort*)(param_2 + 2) <= ip` exits the interpreter when ip reaches
the end of the record. For records with size < 256, reading u16 LE at offset 2
gives the same value as u16 BE at offset 1 — Sacred relies on that aliasing
trick rather than storing size twice.

## Split-VM architecture

`return local_114` (line 2469): the dispatcher returns the opcode value itself to
the caller. So FUN_00472bc0 is **not** the action executor — it's an **operand
decoder**. Each caller has its own per-opcode action handlers, dispatched by
`if (rv == 0x01) ... else if (rv == 0x09) ...`.

```
outer (per-record):
   tag → dispatch to one of 10 subsystem callers
   subsystem caller:
       while (opcode = FUN_00472bc0(&ip, record)) != 0:
           handle_per_caller_action(opcode, ctx);
```

The dialog-dispatcher cases (0x01, 0x29, 0x63, 0x68, 0x69, 0x6A, 0x9D) all
execute the same parsing body — they decode `[cstr1][cstr2]` and store the
resolved data in context. The opcode VALUE is what the caller's handler then
switches on, so `0x01` to subsystem A means a different action than `0x01` to
subsystem B.

## Universal record structure

```
[tag:u8][size:u16 BE][flags:u8]
  [0x01][var_name\0][target_type\0][padding]     ← SETUP: name + target
  ... maybe more 0x01 setups, or other state-setters ...
  [0xNN][action operands...]                     ← ACTION (caller-specific)
  [0x00]                                          ← END
```

- `tag` byte → routes to one of 10 subsystem callers
- `0x01` opcode = "giving you data: variable_name + target_resolver"
- `target_type` values seen:
  - `DLGNPC` — current dialog NPC
  - `QUESTFELLOW` — quest companion
  - `QUESTNPC` — quest NPC
  - `CPOS:hero` — hero position
  - `CPOS:RES:<name>` — position of named resource
  - `res:<name>` — load a Dialog resource by name
- Other opcodes = subsystem-specific actions

Magic constants resolved (full list in `sdk/tools/interpreter_strings.py`
output, 59 entries) include all ECS_* spell names (`ECS_FIREBALL`,
`ECS_HEALING`, `ECS_TELEPORT`, etc.) — 30+ spell identifiers, the verbs Sacred
quests use to invoke creature abilities.

## What's left for full decode

1. The 69 "stub" opcodes (0x06..0xA1, group 6): explained — they are
   dispatch-table indices that the CALLER reads via its handler chain. The
   dispatcher advances ip and returns the opcode; the caller's `if (rv == N)`
   block does the work. Not no-ops; the operand of a caller-level switch.
2. Map record `tag` → subsystem caller. 10 distinct callers, each with a unique
   opcode signature. Matching a known record's tag behaviour to a caller's
   opcode-handler pattern gives the routing table. ~2 hours of Ghidra-side xref
   work.
3. Decode the 0x01 setup parameter encoding fully. The `0x01` op's
   variable_name + target_type strings are read; what comes after them varies by
   target_type:
   - `CPOS:hero` → 3 ints (x, y, z) from context vars
   - `CPOS:RES:<resname>` → also 3 ints from resolved resource
   - `DLGNPC` etc. → just a creature ref

For text-only quest mods (the 14/15 workflow), none of this is needed — full
quest text is already resolved. This interpreter understanding unlocks logic
mods: edit when/how a quest triggers, change reward amounts, add objectives.
